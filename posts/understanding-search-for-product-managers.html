Search for Product Managers

The core function of search is helping your users find something (usually defined as a "document" within an "index"). As a Product Manager solving search problems its important to understanding the links between your documents, your queries, and your users.

User Understanding

Let's assume that you are a Product Manager for a job search engine and your target users are Software Engineers who are looking for their next job. What are their use cases of your product? Software Engineers might search using queries that contain a job title, their desired location, or maybe their skillset in order to find their next job. What if your users don't contain this information in their query? How might you extract that information from your users? If you can learn this information from them (perhaps by allowing them to create a user profile) then you can also use this information to surface relevant documents.

You could use this information to prioritize surfacing certain documents for your users:

  • - If your user is located in Austin, TX then you could prioritize surfacing documents you retrieve to those where "city: austin; state: tx"
  • - If your user has no prior work history then you could prioritize surfacing documents where "experience_level: entry-level"
  • - If your users has python skills then you could prioritize surfacing documents where "skills: python"
  • - If your users has a history of employment in small businesses then you could prioritize surfacing documents where "size: small business"

Document Understanding

The document is what your users are going to be looking for, which will help them achieve their goal. In this scenario: a job posting will help your target user find their next job. You should spend time understanding these documents. Sometimes they're structured and sometimes they're unstructured. What structured information do they contain today? What information is missing that would help your target user achieve their goal? For example, a job posting might contain a job title, company name, location, and job description, but what if your users are searching for a job within an industry? How might you extract that information from the job posting you have available?

You could use the information you have to classify the document:

  • - category: engineering, marketing
  • - industry: finance, healthcare
  • - type: full-time, part-time, contract, internship
  • - experience_level: entry-level, mid-level, senior-level
  • - location_type: on-site, remote, hybrid

You could annotate the document by extracting individual strings:

  • - skills: python, java, agile methodologies
  • - size: small business, medium-sized company, large enterprise
  • - salary_range: low, medium, high
  • - benefits: health insurance, 401k match, relocation assistance

Query Understanding

The query is is how your users are looking for the documents you have so that they can achieve their goal. These queries will represent the use case of your users. You should spend time understanding how your users search so that you can target the right actions to take. How do your users look for jobs? Do they search using job titles, company names, locations, or skills? How might you extract this information from their queries?

You could classify their query:

  • - The query "remote software engineer" can be classified as "location_type: remote; category: engineering"
  • - The query "healthcare marketing" can be classified as "industry: healthcare; category: marketing"
  • - The query "entry-level finance" can be classified as "experience_level: entry-level; industry: finance"

You could annotate individual strings within their query:

  • - The query "product manager google" can be annotated as "job_title: product manager; company_name: google"
  • - The query "python developer" can be annotated as "skills: python; job_title: developer"
  • - The query "$100,000 jobs" can be annotated as "salary_range: high"
  • - The query "no experience jobs" can be annotated as "experience_level: entry-level"

You could even rewrite their query:

  • - The query "product management" can be rewritten as "product manager"
  • - The query "swe" can be rewritten as "software engineer"
  • - The query "twitter" can be rewritten as "x"
  • - The query "20 hour per week jobs" can be rewritten as "part-time jobs"

Retrieval

Using the information you've learned above you can now build your own query to match against the documents in your index. For example, if you know that your user has python skills and searches for "remote swe" then a query you might build to retrieve documents could be:

{
    "query": {
        "bool": {
        "must": [
            {
            "multi_match": {
                "query": "software engineer",
                "fields": ["job_title^3", "job_description"]
            }
            },
            {
            "term": {
                "location_type": "remote"
            }
            },
            {
            "term": {
                "skills": "python"
            }
            }
        ],
        "should": [
            {
            "match": {
                "job_title": "swe"
            }
            }
        ]
        }
    },
    "sort": [
        {
        "_score": {
            "order": "desc"
        }
        }
    ]
    }

This query represents the intent of your user. Using this query you can retrieve a list of relevant documents.

Ranking

You can rerank the list of documents you've retrieved so that the most relevant documents are in the top positions of the search results. If your user searches for "product manager jobs austin tx" they may want to see popular jobs that other users have already applied to. If your business goal is to increase applies then you may want to increase the visibility of the retrieved job postings that users have previously applied to. The documents can be reranked with a ranking model that predicts a relevance score using a set of selected features from the documents, the query, and the user.

Ranking documents relies on the user's query and its relationship to the document. We call these query-dependent features:

  • - BM25 score of the query and job posting
  • - Cosine similarity between the query and job posting embeddings
  • - Jaccard similarity between the query and fields of the job posting
  • - Apply rate of a job posting for the query
  • - Explicit feedback signals, such as a thumbs down, of a job posting for the query

Ranking documents also relies on the retrieved document itself. We call these query-independent features:

  • - Click-through rate (CTR) of the job posting
  • - Total applies to the job posting
  • - Creation date (freshness) of the job posting
  • - Quality of the job posting
  • - Brand authority of the job posting company
  • - Rating of the job posting company

Additionally, ranking documents can also rely on the context of the user. We call these contextual ranking features:

  • - Location of the user: If a user is based in Austin, TX then job postings in Austin, TX may rank higher.
  • - Preferences of the user: If a user prefers to work remotely then job postings that are remote may rank higher.
  • - Search history of the user: If a user frequently searches for "software engineer internships" then job postings with "job_type": "internship" may rank higher.
  • - Engagement history of the user: If a user frequently applies to job postings with "skills": "python" and job postings with "salary_range": "high" then job postings that match both of these criteria may rank higher.

With all of these features combined and the output of your ranking model you should now have a ranked list of documents that are relevant to your user and their query that will help your user achieve their goal of finding their next job.
